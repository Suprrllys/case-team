## 1. Понимание кейса

### Постановка (своими словами)

Подать проект «Команда ИИ-агентов для решения бизнес-кейсов» (`/case-team`) в конкурс Practicum Award 2026 от Яндекс Практикума, в номинацию **«Техно-продуктовый вклад»**.

### Цели

- **Базовый успех:** пройти отбор финалистов (попасть в шорт-лист)
- **Цель:** финалист номинации
- **Растяжка:** приз номинации или спецприз «Будущее за ним» (для опыта до 2 лет — автор подходит, PM с дек 2025)

### Горизонт, ресурсы, ограничения

- **Дедлайн подачи:** 15 мая 2026 (завтра)
- **Финал:** 16 июля 2026, Экстрополис, Москва (потенциальное живое интервью с жюри ~15 мин)
- **Бюджет:** 0 ₽
- **Команда:** 1 человек (автор-pm) + команда `/case-team` агентов
- **Подача:** один участник, остальные участники подтверждают позже (для нас неактуально — sole authorship)
- **Резидент РФ** — выполняется

### Критерии оценки (буквально из лендинга — основа отбора)

Жюри Техно-продуктовой номинации ставит оценку по **4 критериям**:

1. **Техническая реализация** — качество AI в продукте, обоснованность выбора модели, масштабируемость, производительность
2. **Польза и эффект** — KPI и целевые метрики, верификация результатов, социальная и/или экономическая значимость, пользовательская ценность
3. **Нестандартный подход в применении ИИ** — новизна идеи, оригинальность инструмента, визуальное решение
4. **Качество презентации** — структура и логика

### Контекстные критерии (мои гипотезы — не явные требования)

- «Реальная польза бизнесу» в формулировке самого конкурса → **экономическая значимость в рублях весомее социальной**
- Жюри техно-практики (CTO Практикума, AI Studio, LetAI, ФАНС, Замесин) → **им важна архитектура + обоснованность выбора + измеримый эффект**, не маркетинговая упаковка
- «Опыт до 5 лет» → жюри не ждёт зрелого SaaS, важны **потенциал** и качество продумывания

### Подсказки в материалах

- На лендинге упомянуты «инструменты написания кода с помощью ИИ» как пример — наш проект шире (методологические AI-сервисы, не только кодинг). Это можно подсветить как расширение категории.
- Видео до 5 мин на Я.Диск + SourceCraft-репо — опциональные усиления; решим по времени

### Ограничение от пользователя

**До 10 основных слайдов** (включая титул и команду). Остальное — в приложения для Q&A.

## 2. Концепция решения

**Альтернатива стратегическому консалтингу через мультиагентную AI-команду.**

`/case-team` — система специализированных Claude-агентов с разделением ответственности, которая выполняет **3 Core Jobs** клиента-фаундера: подготовка пакета для питча инвестору / стратегический ответ на изменение рынка / go/no-go новой инициативы. На выходе — **физические рабочие файлы** (`.xlsx` финмодель, `.pptx` презентация, `.md` Decision Log), а не текст в чате.

**Формула концепции:** Value Proposition (продуктовый запуск) — «какой работе помогаем, для кого, чем лучше альтернатив».

**Тип концепции (по §8.8 case-method):** сингулярная — одна центральная ставка («альтернатива консалтингу») с поддерживающими слоями (архитектура / экономика / эталонная база). Не ансамблевая — у нас нет нескольких параллельных инициатив под разные барьеры.

### Иерархия работ клиента (AJTBD) — версия после уточнения пользователя

> **Поправка:** ранее я ошибочно записал «3 Core Jobs» (питч / реакция / новый продукт). На самом деле это **контексты использования одной Core Job**. Правильная иерархия ниже.

**Big Job клиента:**
> Принимать сильные стратегические решения в моменте и проводить их через стейкхолдеров, чтобы развивать бизнес и не упускать окна возможностей.

**Core Job (одна, выполняется `/case-team` ПОЛНОСТЬЮ — точка покупки):**
> Подготовить полный набор материалов решения бизнес-кейса (презентация + финмодель + анализ + Decision Log + позиционирование).

**Контексты использования Core Job (когда клиент нуждается в этой работе):**
- Питч инвестору
- Реакция на изменение рынка
- Запуск нового продукта или выход на новый рынок
- Защита решения перед советом директоров / стейкхолдерами
- Любая ситуация требующая структурированной стратегической подачи

**Micro Jobs (на уровень ниже Core Job — входят в неё, не покупаются по отдельности):**
- Презентация (.pptx с навигацией и pre-flight check)
- Финмодель (.xlsx с формулами и sensitivity)
- Анализ рынка / конкурентов / ЦА
- Концепция и позиционирование
- Decision Log с обоснованием решений

**Small Jobs (на том же уровне, что Core Job, `/case-team` НЕ выполняет — остаётся за человеком):**
- Принять финальное go/no-go решение
- Защитить решение лично перед инвестором/советом (живая коммуникация)
- Внедрить стратегию операционно
- Договориться с конкретными партнёрами / нанять людей

**Почему такая иерархия сильнее:**
1. **Одна Core Job легче продаётся** — клиент покупает «полный набор материалов», не 3 разных работы
2. **Micro Jobs выводятся как физические артефакты** на выходе — это и есть наш ключевой USP («физические файлы, а не текст в чате»)
3. **Контексты использования** показывают разные сценарии, расширяют применимость без размывания продукта
4. **Граница ответственности явная** через Small Jobs — это усиливает доверие жюри (мы не обещаем магию)

## 4. Decision Log

### D1. Выбор номинации
- **Решение:** «Техно-продуктовый вклад» (не «Креативное решение»)
- **Почему:** проект — мультиагентная AI-система для автоматизации интеллектуальной работы. Жюри Техно — техно-практики; им важна архитектура + измеримый эффект. Креативное — для дизайна/копирайтинга/визуала, что не основной фокус.

### D2. Выбор ЦА (подтверждено пользователем)
- **Решение:** B2B основная (стартапы / малый и средний бизнес / корп. L&D / солопренёры-консультанты), Edu вторичная (студенты кейс-чемпионатов + MBA)
- **Почему:** жюри Practicum в формулировках акцентирует «реальная польза бизнесу» → экономическая значимость в рублях весомее. Edu остаётся как social proof + demand generation. Не моно-B2B (без edu) — теряем социальную значимость; не моно-edu — теряем экономическую значимость.

### D3. Тип концепции
- **Решение:** Сингулярная («альтернатива консалтингу»)
- **Почему:** проект имеет одну центральную ставку и поддерживающие слои. Нет 3-4 разнородных барьеров ЦА, которые требовали бы параллельных инициатив. По §8.8 case-method это сингулярный тип (как Axenix), не ансамблевый (как Inno).

### D4. Структура презентации (под лимит 10 основных слайдов)
- **Решение:** 8 основных контентных слайдов (Cover → Exec Summary → Анализ → Концепция → Реализация → Эффект → Команда → Финал) + приложения для Q&A
- **Почему:** структура задана пользователем; классическая логика «контекст → решение → как → эффект». Каждый слайд закрывает ≥1 критерий жюри. Детальное (AJTBD-персона, конкурентный анализ, технологический стек, детальная финмодель, roadmap) — в приложения как страховка для Q&A.

### D5. Volna 1 research — переиспользуем v1
- **Решение:** Файл `analyst/practicum-award-context.md` от v1 остаётся как валидный research (Practicum Award первый год, конкуренты в multi-agent space, рынок консалтинга РФ, 3 бенчмарка AI-экономии — всё актуально)
- **Почему:** факты не зависят от ошибок в нашей концепции. Перезапускать аналитика — потеря времени без новой информации.

### D6. Финмодель — переиспользуем v1
- **Решение:** Файл `financier/unit-economics.md` от v1 остаётся как валидная финмодель (90% экономия, sensitivity 84-98%, TAM 266k компаний)
- **Почему:** цифры объективные, не зависят от формулировки концепции. Главные якоря — реальные ставки консалтеров РФ + расход токенов Claude API + реальный baseline пользователя (40 и 140 чел.-часов на Inno и Cup Moscow).

### D7. Позиционирование — переделать под правильную концепцию
- **Решение:** Перезапустить `case-marketer` с правильной концепцией (AJTBD-иерархия работ + Mot-Moment персона; промпты Замесина как **инструмент агента**, не «методология команды»)
- **Почему:** v1-positioning был сделан под ошибочное позиционирование «AJTBD = методология бренда». Это нужно убрать — это инструмент case-marketer наравне с другими, не наш бренд.

### D8. Workflow агентов (на слайд «Реализация»)
- **Решение:** Показать **реальный** workflow: менеджер↔специалисты через файлы; cross-agent через blockers/ всегда через менеджера; YAML-контракт только в финальной миле менеджер→дизайнер
- **Почему:** в v1 я ошибочно подал «YAML-handoff между всеми». Реально YAML — только последний шаг «контент→формат». Между специалистами — обычная файловая коммуникация. Это важно для жюри (техно-практики поймут детали).

### D10. Модель монетизации — `/case-team` встроен в Claude подписку, не отдельный продукт
- **Решение:** Не подавать `/case-team` как продукт с собственной ценой. Это **open-source skill для Claude Code**, который пользователь устанавливает один раз в свою среду. **Стоимость для клиента — $0 дополнительно**, если у него есть Claude подписка ($20-100/мес уже оплачена для других задач).
- **Что меняется в подаче:**
  - **Slide 2:** «по стоимости подписки на нейросеть» — буквально правда, не аналогия
  - **Slide 6:** реальная экономия для Claude-юзера **>99%** vs in-house (99k → $0); для новичка **до 95-99%** vs in-house с учётом покупки подписки ($20-100/мес vs 99k/кейс)
  - **Slide 7:** барьер «цены/риска» из 5 барьеров **снят полностью** — нет отдельной транзакции
  - **Slide 9 (слоган):** работает на «нулевую транзакцию для существующих юзеров» — это категориальный сдвиг, не price war
- **Источник цифр:** Claude Pro ~$20/мес, Claude Max ~$100/мес (Anthropic pricing 2026); ставки in-house из `financier/unit-economics.md` без изменений
- **Что НЕ меняется:** финмодель `unit-economics.md` валидна — там посчитана **стоимость работы агентов** (5 250 ₽ время менеджера + 5 680 ₽ токены). Это **внутренний расход на кейс**, не цена для клиента. Для клиента дополнительный расход = 0
- **Почему это сильнее обычной «низкой цены»:** жюри-практики увидят, что это **категориальный сдвиг** (стратегический консалт встроен в инструмент общего пользования), не маркетинговый трюк. Это снимает риск восприятия «90% экономия слишком красиво — где обман»

### D9. YAML — не USP, а внутренняя техника реализации
- **Решение:** **Не подавать YAML-контракт как «крутое архитектурное решение» в подаче.** Это межагентный файл внутри одного запуска `/case-team`. Пользователь его не видит и ничего с ним не делает в regular-режиме.
  - **Regular-режим:** всё происходит внутри запуска `/case-team` — пользователь получает готовый `.pptx`. YAML существует только как промежуточный файл между case-manager и case-designer; пользователь его не трогает.
  - **Premium-режим:** пользователь сейчас вручную загружает YAML в Claude Design в браузере → но это **временная мера**. После интеграции Claude Design в десктоп-приложение Claude и этот шаг автоматизируется.
- **Что подавать в кейсе как USP вместо «YAML-handoff»:**
  - Реальные USP — **5 ролей с разделением «куратор смысла vs конвертер»**, **дотренировка на 2 реальных эталонах**, **иерархия источников методологии**, **физические артефакты на выходе** (xlsx + pptx + md, не текст в чате), **мета-демо**.
  - YAML упоминается только как технический штрих на слайде «Реализация» — в схеме workflow один из 4 типов коммуникации между ролями.
- **Почему:** жюри техно-практики увидят, что «YAML между двумя агентами» — не уникальное решение, это просто структурированная передача данных. Подавать это как ключевую фичу = снижать доверие. Реальные USP сильнее.
